# Authentication

The MCP server has **two trust boundaries** that never mix:

```text
   AI Client                                       Zammad API
       │                                                ▲
       │  (1) OAuth 2.1 + PKCE                         │  (2) Bearer token
       │      against MCP server                       │      forwarded
       ▼                                                │
  MCP server  ─────────  AuthMode dispatch  ──────────┘
                                  │
                  ┌───────────────┼───────────────┐
                  │               │               │
              zammad           oidc             none
            (PRIMARY)        (external IdP)    (dev only)
```

`AUTH_MODE` decides what fills the two boxes:

| `AUTH_MODE` | Inbound (AI client → MCP) | Outbound (MCP → Zammad) | Role enforcement |
| --- | --- | --- | --- |
| **`zammad`** | OAuth 2.1 against Zammad's `/oauth/*` endpoints | The user's Zammad access token is forwarded (user-context) | `MCP_ALLOWED_ROLES` enforced + Zammad's own permissions |
| `oidc` | OAuth 2.1 against an external IdP (Entra, Keycloak, ...) | Static `ZAMMAD_API_TOKEN` (single Zammad identity) | Zammad's own permissions only (allowlist is a no-op) |
| `none` | none — endpoint is unprotected | Static `ZAMMAD_API_TOKEN` | Zammad's own permissions only |

`zammad` is the recommended mode because it is the only one where Zammad
sees who the actual MCP user is.

## Mode 1 — Zammad as OAuth2 provider (recommended)

This uses Zammad's built-in **OAuth2 Applications** feature, which ships in
**Zammad v6.0 and later**. The Zammad install is the OAuth IdP; the MCP
server is the OAuth client; the human AI-tool user is the resource owner.

### Step 1: Register the application in Zammad

1. Sign in to Zammad as an administrator.
2. Open the admin sidebar: **Manage → OAuth2 Applications**.
3. Click **+ Add**.
4. Fill in the dialog:

   | Field | Value |
   | --- | --- |
   | **Name** | `BAUER GROUP MCP` (any human-friendly label) |
   | **Redirect URI** | `${PUBLIC_BASE_URL}/auth/callback`  ← must match exactly |
   | **Scopes** | `read write` |
   | **Confidential** | Yes |

5. **Save.** Zammad shows the generated **Client ID** and **Client Secret**
   on the next screen — copy them now; the secret cannot be retrieved later.

> **Important:** the redirect URI must exactly match the value Zammad has on
> file, including scheme (`https://`) and trailing slash (none).

### Step 2: Configure the MCP server

Fill in `.env`:

```env
AUTH_MODE=zammad
PUBLIC_BASE_URL=https://zammad-mcp.example.com

ZAMMAD_URL=https://support.example.com
ZAMMAD_OAUTH_CLIENT_ID=<copied from Zammad>
ZAMMAD_OAUTH_CLIENT_SECRET=<copied from Zammad>
ZAMMAD_OAUTH_SCOPES=read write

# Required - never empty in production
AUTH_JWT_SIGNING_KEY=<generated by scripts/generate-env.py>

# Role allowlist (covered in role-based-access.md)
MCP_ALLOWED_ROLES=Admin,Agent
```

Redeploy. The first AI client that connects will be redirected to Zammad
for login; after consent, Zammad issues an access token that the MCP server
forwards to every Zammad API call for that session.

### How user-context works

1. AI client opens the MCP `/mcp` endpoint and hits the auth wall.
2. MCP server redirects the client to its own `/authorize` endpoint.
3. That endpoint redirects to `${ZAMMAD_URL}/oauth/authorize`.
4. The user logs in to Zammad and consents to the requested scopes.
5. Zammad redirects back to `${PUBLIC_BASE_URL}/auth/callback` with an
   authorization code.
6. The MCP server exchanges the code for a Zammad access token at
   `${ZAMMAD_URL}/oauth/token`.
7. The MCP server validates the token by calling `/api/v1/users/me` —
   the response carries the user's role set.
8. The MCP server issues its **own** JWT to the AI client (signed with
   `AUTH_JWT_SIGNING_KEY`). The Zammad access token is stored encrypted,
   keyed by the JWT's `jti`.
9. When the AI client calls a tool, the MCP server retrieves the Zammad
   access token from storage and forwards it as
   `Authorization: Bearer <token>` to Zammad. Zammad sees the call as
   coming from the authenticated user, and its own permission system
   gates which resources are visible.

The Zammad access token never leaves the MCP server, and the AI client
never sees a token that Zammad would accept directly.

### Token lifetime

Zammad's OAuth2 access tokens default to expiring after 2 hours. The MCP
server stores the refresh token alongside the access token and rotates it
transparently. If the user revokes the Zammad OAuth grant (Profile →
Token Access → Revoke), the next refresh attempt fails and the MCP client
is forced to re-authenticate.

## Mode 2 — External OIDC IdP

Use this when you already have SSO via Entra ID, Keycloak, Authentik,
Zitadel, Auth0, or Okta and want the same identity provider to gate the
MCP. Or when your Zammad pre-dates v6 and lacks the OAuth2-Applications UI.

**Trade-off:** Every MCP call from every user runs against Zammad as
**one shared identity** (the owner of `ZAMMAD_API_TOKEN`). Zammad cannot
distinguish "Agent Alice asked the AI to delete ticket #42" from "the
service account did it." If that matters for audit / compliance, use Mode 1.

### Step 1: Register the OIDC client

Depends on your IdP. The redirect URI to register is
`${PUBLIC_BASE_URL}/auth/callback`.

### Step 2: Get a Zammad Personal Access Token

1. Sign in to Zammad as the user / service account whose permissions
   should apply.
2. Click your avatar → **Profile → Token Access**.
3. Click **+ Create**, choose appropriate permissions
   (`ticket.agent`, `user.reader`, etc.), and save.
4. Copy the displayed token — it is shown only once.

### Step 3: Configure the MCP server

```env
AUTH_MODE=oidc

# OIDC IdP (use discovery if available)
OIDC_DISCOVERY_URL=https://idp.example.com/.well-known/openid-configuration
OIDC_CLIENT_ID=<from IdP>
OIDC_CLIENT_SECRET=<from IdP>
OIDC_SCOPES=openid profile email

# Static Zammad credentials - REQUIRED in oidc mode
ZAMMAD_URL=https://support.example.com
ZAMMAD_API_TOKEN=<the personal access token from step 2>
```

The role allowlist (`MCP_ALLOWED_ROLES`) is a **no-op** in this mode
because external OIDC tokens carry no Zammad role claims. Zammad still
enforces its own permission set on every API call.

## Mode 3 — No auth (development only)

```env
ENVIRONMENT=development
AUTH_MODE=none
ZAMMAD_URL=https://zammad-dev.local
ZAMMAD_API_TOKEN=<Zammad PAT>
```

The container refuses to start in any other environment. Use only for
local iteration; never expose the resulting endpoint on a network.

## Switching between modes

A mode switch invalidates any active OAuth state because the inbound IdP
changes. Operators should:

1. Update the `.env` and restart the container.
2. Have AI clients re-run their connector setup (Settings → Connectors →
   reconfigure). Existing sessions silently fail on the next token
   refresh; the client UI will prompt for a re-login.
3. Optionally clear the OAuth state store
   (`docker exec bg-zammad-mcp-redis redis-cli FLUSHDB`, or
   `docker volume rm bg-zammad-mcp-oauth` for disk-backed deployments).
   This forces every client to re-register via DCR.

## Frequently-asked questions

**Q: Can I run AUTH_MODE=zammad against a Zammad behind HTTP basic auth?**
A: No. The MCP server cannot inject Basic credentials into the OAuth
authorize redirect that the AI client follows. The browser would see the
basic-auth prompt and fail.

**Q: My Zammad is at a sub-path (`https://intranet.example.com/zammad`).**
A: Set `ZAMMAD_URL=https://intranet.example.com/zammad` and adjust
`ZAMMAD_OAUTH_AUTHORIZE_PATH=/oauth/authorize`,
`ZAMMAD_OAUTH_TOKEN_PATH=/oauth/token`,
`ZAMMAD_USERINFO_PATH=/api/v1/users/me` to keep them sub-path-relative.
The MCP server concatenates them onto `ZAMMAD_URL`.

**Q: Does the MCP server cache /users/me responses?**
A: Yes, for `MCP_ROLE_CACHE_TTL_SECONDS` seconds (default 60). Role
changes propagate within that window. Set to `0` to disable caching.

**Q: Can I use both AUTH_MODE=zammad and a fallback ZAMMAD_API_TOKEN?**
A: Yes. If the upstream-token lookup fails (storage corruption, expired
token, etc.) and `ZAMMAD_API_TOKEN` is set, the call falls back to the
static token. A warning is logged. This is defensive — set the token in
zammad-mode only when you have a clear use-case for it.
