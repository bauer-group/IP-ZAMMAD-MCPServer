# Installation

The Zammad MCP Server ships as a single multi-arch Docker image
(`linux/amd64` + `linux/arm64`) plus three reference `docker-compose` files
covering local development, self-hosted Traefik, and Coolify-managed
deployments. All three pull the same image and read the same `.env` file —
the differences are purely about how requests reach the container.

## Prerequisites

| Component | Minimum | Notes |
| --- | --- | --- |
| Docker Engine | 24.x | Compose v2 plugin recommended |
| Self-hosted **Zammad** | **v6.0** | v7.x fully supported. v5.x and earlier lack the OAuth2-Applications UI - use `AUTH_MODE=oidc` + a Personal Access Token. |
| Public hostname & DNS | yes (prod) | Lets-Encrypt resolves it. Skip for `AUTH_MODE=none` dev runs. |
| Reverse proxy | Traefik 2/3 (prod) | Coolify ships its own. Local dev runs without one. |
| Redis 7.x / 8.x | recommended | OAuth state store. Disk-fallback works without it but is single-node only. |

Generating the secrets is one command:

```bash
python scripts/generate-env.py
```

This writes a `.env` file with fresh values for `AUTH_JWT_SIGNING_KEY` and
`AUTH_STORAGE_ENCRYPTION_KEY`. You then fill in the Zammad-specific
fields manually (see [authentication.md](./authentication.md)).

## Path A — Local Development

Use this to iterate on changes without setting up TLS or an IdP.

```bash
cp .env.example .env
python scripts/generate-env.py --force

# Edit .env:
#   ZAMMAD_URL=https://your-zammad.example.com
#   ZAMMAD_API_TOKEN=<a Zammad Personal Access Token>
#   AUTH_MODE=none      # dev only
#   ENVIRONMENT=development

docker compose -f docker-compose.development.yml up -d --build
```

The MCP server is now reachable at `http://localhost:8000/mcp`. The
landing page at `http://localhost:8000/` shows the live system status,
the configured Zammad backend, and a copy-paste-ready MCP URL.

**Caveats of `AUTH_MODE=none`:**

- The endpoint is **unprotected** — anyone who can reach `localhost:8000`
  can list every ticket the API token has access to.
- The container refuses to start with `AUTH_MODE=none` and
  `ENVIRONMENT=production`.
- All API calls share the single configured `ZAMMAD_API_TOKEN`; there is
  no per-user role enforcement.

For LAN testing with real OAuth see Path B with `ENVIRONMENT=staging`.

## Path B — Self-hosted with Traefik

Recommended for an internal MCP behind your own Let's-Encrypt-enabled
Traefik. The compose file expects a pre-existing external network (default
name `EDGEPROXY`).

```bash
docker network ls | grep EDGEPROXY   # must exist; create with `docker network create EDGEPROXY`
cp .env.example .env
python scripts/generate-env.py --force
```

Fill in `.env`:

```env
ENVIRONMENT=production
ZAMMAD_MCP_HOSTNAME=zammad-mcp.example.com
PUBLIC_BASE_URL=https://${ZAMMAD_MCP_HOSTNAME}
PROXY_NETWORK=EDGEPROXY

ZAMMAD_URL=https://support.example.com

AUTH_MODE=zammad
ZAMMAD_OAUTH_CLIENT_ID=<from Zammad OAuth2 Applications>
ZAMMAD_OAUTH_CLIENT_SECRET=<from Zammad OAuth2 Applications>

MCP_ALLOWED_ROLES=Admin,Agent
```

Then:

```bash
docker compose -f docker-compose.traefik.yml up -d
```

Traefik picks up the labels, requests a certificate, and routes
`https://${ZAMMAD_MCP_HOSTNAME}/mcp` to the container. Verify with:

```bash
curl -fsSI https://${ZAMMAD_MCP_HOSTNAME}/healthz
```

## Path C — Coolify

Coolify treats the compose file as an application. The bundled Traefik
handles TLS, DNS, and certificate renewal.

1. **Create application** — Type: `Docker Compose`. Paste
   `docker-compose.coolify.yml` (or point at this repo).
2. **Domains** — Add `zammad-mcp.example.com`. Coolify auto-populates
   `SERVICE_FQDN_ZAMMADMCP` and wires Traefik.
3. **Environment Variables** — Paste in the pre-shared values from your
   `.env`. The crypto material (`AUTH_JWT_SIGNING_KEY`,
   `AUTH_STORAGE_ENCRYPTION_KEY`) is auto-generated by Coolify on first
   deploy.
4. **Deploy.**

Coolify-specific notes:

- The compose file **deliberately omits a custom `networks:` block**. Adding
  one causes Traefik 504 Gateway Timeouts because Coolify attaches services
  to two networks and Traefik picks non-deterministically. Service-name
  DNS between `zammad-mcp` and `redis` still works via Coolify's
  auto-created bridge.
- Use the Coolify "Persistent Storage" UI to attach a volume to
  `/app/data/oauth-storage` if you don't use Redis. Without it, OAuth state
  is wiped on every redeploy.

## What's running

After any of the three paths, the container exposes:

| Path | Purpose |
| --- | --- |
| `POST /mcp` | The MCP JSON-RPC endpoint (OAuth-gated) |
| `GET /` | Human-readable status & quickstart page |
| `GET /healthz` | Liveness probe for orchestrators |
| `GET /logo.svg` | Brand asset for the OAuth consent screen |
| `GET /.well-known/oauth-protected-resource/mcp` | RFC 9728 metadata |
| `GET /.well-known/oauth-authorization-server` | OAuth Server metadata |

Inside the container:

```bash
docker exec -it bg-zammad-mcp python src/main.py health    # probes Zammad
docker exec -it bg-zammad-mcp python src/main.py probe     # detects v6/v7
docker exec -it bg-zammad-mcp python src/main.py tools     # lists MCP tools
```
